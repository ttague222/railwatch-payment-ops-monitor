# AI-First Product Owner — Take-Home Assessment

## Overview

This is an async, spec-driven product exercise designed to evaluate how you think, design, and build using AI-first workflows. You will identify a problem worth solving, define requirements, design a system, and implement a working application using Kiro.

**Time expectation:** 4–6 hours (not timed — complete at your own pace within 5 business days)
**Deliverable:** A public GitHub repository containing your complete project

---

## The Challenge

Build something useful in the personal finance or fintech space.

That's it. You decide what to build, who it's for, and why it matters.

### Constraints

1. **Domain:** Your application must be relevant to personal finance, banking, or fintech. It should solve a real problem a real user might have.

2. **External API integrations:** You must integrate at least 2 public APIs. Some examples you can use include:

   - [Frankfurter API](https://frankfurter.dev/) — Real-time and historical currency exchange rates (no auth required)
   - [Alpha Vantage](https://www.alphavantage.co/) — Stock quotes, time series, technical indicators (free API key, instant signup)
   - [Marketaux](https://www.marketaux.com/) — Financial news with sentiment analysis (free tier, instant signup)
   - [ExchangeRate-API](https://www.exchangerate-api.com/docs/free-exchange-rate-api) — Currency conversion rates (no auth required)

   These are suggestions — you are free to use any public APIs that serve your product vision.

3. **Built with Kiro:** The entire project must be built using Kiro's spec-driven workflow. If you haven't used Kiro before, that's fine — figuring it out is part of the exercise.

4. **Published to GitHub:** Your completed project must be in a public GitHub repository. The repo must include everything — source code, Kiro spec artifacts (`.kiro/` directory), your AI collaboration context, and README. Do not gitignore the `.kiro/` folder. We will review the full repo history.

---

## What You Must Deliver

Your GitHub repository must contain:

### 1. Kiro Spec Artifacts

Your project must include the `.kiro/specs/` directory generated through Kiro's spec workflow:
- `requirements.md` — User stories and acceptance criteria
- `design.md` — Technical design, API integration patterns, data flow
- `tasks.md` — Implementation task list

We will review the spec quality as heavily as the code. These should reflect your product thinking — not just a restatement of this brief.

### 2. AI Collaboration Context

We need to understand how you worked with AI throughout this exercise. Your repo must include context that shows your thought process, decisions, and iterations. How you capture and organize that context is up to you.

### 3. A Working Application

It must run. Technology stack is your choice. Creativity in how you present the experience matters — we're evaluating product thinking, technical fluency, and how you bring an idea to life.

### 4. A README.md

- What you built and why — what problem does it solve? Who is the user?
- Which APIs you integrated and how they serve the product
- How to run it locally
- Product decisions you made and your reasoning
- What you would change or add with more time

## What We're Evaluating

| Dimension | What We're Looking For | Weight |
|-----------|----------------------|--------|
| **Product Thinking** | Did you identify a real problem? Can you articulate who the user is and why this matters? Are your requirements driven by user needs, not just technical capabilities? | High |
| **Spec Quality** | Are the requirements clear, testable, and well-structured? Does the design show real technical thinking — data flow, error handling, API patterns, edge cases? Are tasks logical and incremental? | High |
| **Technical Fluency** | Does the code work? Do you handle API errors, loading states, and edge cases? Is the integration pattern clean? Can you read and understand what Kiro generated? | High |
| **AI Collaboration** | How effectively did you use Kiro? Did you iterate on the problem and specs before jumping to code? Does the context you captured show thoughtful collaboration? | Medium |
| **Ambiguity Navigation** | We intentionally left the problem open-ended. How did you handle that? Did you make decisions and move forward, or did you get stuck waiting for direction? | Medium |
| **Code Quality** | Is the code organized, readable, and maintainable? We don't expect production polish, but we expect someone who understands what good looks like. | Low |

---

## Tips

- **Start with the problem, not the solution.** The best submissions have a clear point of view on the user and the problem before any code is written.
- **Show your thinking.** Your repo should tell the story of how you got to the final result — not just the final result.
- **Scope is a product skill.** Knowing what to build and what to cut matters as much as execution.
